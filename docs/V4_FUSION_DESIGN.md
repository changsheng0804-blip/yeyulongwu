# V4 融合设计: 结构之骨 × 生成之血

## 核心命题

V3 做对了一件事: 把音乐从"生成"升级为"编排"——鼓的三件套、精确的十六分音符网格、四乐章起承转合。
V2 做对了另一件事: 让图形保持在临界态——不是僵死结构也不是噪音，是"将具未具"的张力。

**融合不是折中。是在 V3 的骨架上，让 V2 的活肉自己长出来。**

## 二、四大融合策略

### 策略 A: 旋律——骨架定、血肉活

**现状**
- V3: 旋律每个音都写死在总谱里 (`mel:['5---4-5-7---5---',...]`)
- V2: 旋律完全由粒子状态实时决定,没有结构约束

**融合方案**
旋律分两层:
1. **骨架音** (V3 写定): 每个小节第一拍的主旋律音,决定方向
2. **填充音** (V2 生成): 骨架音之间的音由级进生成器自由填充

```js
// 记谱法扩展: '>' 标记骨架音,其余由生成器填充
mel: ['>5--4-5-?---?---','?---?>---?-?---',...];  
//  骨架: 第1音是 G(5级) → 第5音爬向?由生成器决定
//  生成器逻辑: 在骨架音之间级进填补,受当前和弦约束
```

具体算法:
1. 读取当前小节的骨架音列表 (时间戳 + 音级)
2. 每 16 分音符,如果当前步有骨架音→直接播放
3. 如果没有→V2 级进生成器决定是否发声、发几级:
   - `melodyDeg` 向下一骨架音级进 (`Math.max(-1,Math.min(1,d))`)
   - 出音概率由**和弦协和度 + 粒子动能**联合决定
   - 音符在**当前和弦音内**优先,其次五声音阶

**为什么这样好**
- 保留 V3 的"旋律有方向感"——听众能跟住一条线
- 保留 V2 的"每次不同"——填充音每次播放都不一样
- 和弦约束保证生成的音不会"跑调"

### 策略 B: 物理参数——基准定、活量游

**现状**
- V3: 每乐章写死 spring/swirl/trail 三个值,乐章切换时突变
- V2: 三个值靠蒙特卡洛搜出了临界态区间

**融合方案**
每个乐章定义一个**基准值 + 允许波动范围**,运行时在范围内平滑游走:

```js
const MOVEMENT_PARAMS = [
  { // 第一乐章 · 星涡 · 序曲
    preset: 3,
    spring:  { base: 3.5,  wander: 2.0, rate: 0.08 },  // 基准 3.5, ±2 内缓慢游走
    swirl:   { base: 1.3,  wander: 0.6, rate: 0.12 },
    trail:   { base: 8,    wander: 1,   rate: 0.05 },
  },
  { // 第二乐章 · 鱼龙舞 · 行板
    preset: 0,
    spring:  { base: 6.0,  wander: 2.5, rate: 0.15 },
    swirl:   { base: 0.8,  wander: 0.5, rate: 0.10 },
    trail:   { base: 4,    wander: 1,   rate: 0.08 },
  },
  // ...
];
```

每个参数用正弦游走:
```js
springCur = base + wander * Math.sin(t * rate + phaseSeed);
```

**关键**: 游走范围必须落在蒙特卡洛搜出的"优质区间"内。不在死亡区徘徊。

**为什么这样好**
- 乐章之间的参数不会突变——都是平滑过渡
- 但同一乐章内参数也不是死的——每几秒悄悄变一下
- 听众不会注意到参数在变,但会感觉画面"在呼吸"

### 策略 C: 鼓的视觉角色——保持 V3 设计,加入贝斯维度

**现状**
V3 的三种鼓视觉角色已经设计得很好:
- 底鼓 → 膨胀 + 镜头一震
- 军鼓 → 抖动 + 亮度 + 色相
- 踩镲 → 微光

**融合增强**
新增贝斯的低频视觉响应:

```js
// 贝斯不在画面层做大幅动作,而是在粒子速度上叠加微弱低频振动
// 贝斯发声时: 所有粒子 z 轴速度加一个极小的正弦波
if(bassEnv > 0){
  for(let i = 0; i < N; i++){
    vz[i] += Math.sin(t * 18 + i * 0.01) * bassEnv * 0.03;
  }
}
```

这不是"可见"的动作——是粒子有一丝微不可察的上下浮动,像空气在震。观众感知不到它是来自贝斯,但关掉贝斯后会觉得画面"不对"。

### 策略 D: 乐章过渡——不是切换,是渐变

**现状**
V3 乐章切换时: preset 立即变,音色立即变,物理参数立即变。有割裂感。

**融合方案**
每个乐章结束时启动一个**过渡期**（2 小节 ≈ 4-5 秒）:

1. **形态插值**: 不是 `preset = 新值`,而是**两个图案的公式混合计算目标位置**
   ```js
   function target(i, N, t, mm, blend){
     const gOld = targetPreset(oldPreset, i, N, t, mm);
     const gNew = targetPreset(newPreset, i, N, t, mm);
     return [
       gOld[0] * (1-blend) + gNew[0] * blend,
       gOld[1] * (1-blend) + gNew[1] * blend,
       gOld[2] * (1-blend) + gNew[2] * blend,
     ];
   }
   ```
   粒子会看到一个形态在"融化"成另一个形态——这是纯算法才能做到的效果,其他媒介做不到。

2. **音色过渡**: 旧 wave 和新 wave 各发一半音量,交叉淡入淡出

3. **光迹过渡**: trail 从旧值平滑滑到新值,画面从模糊逐渐清晰(或反向)

4. **过渡期的鼓**: 保留旧乐章的踩镲,提前引入新乐章的底鼓——鼓的过渡做**交叉渐变**而不是硬切

## 四、实现路线图

### 第一阶段: 核心融合 (关键路径)
1. 将 V3 的乐谱格式扩展为支持骨架音标记
2. 实现"骨架音约束下的级进生成器"

## 六、需验证的关键风险

1. **图案混合渲染的视觉质量**: 两个完全不同类型的粒子公式混合,中间态是否好看?还是会出现"面条"?
   - 缓解: 只混合同类型图案 (星涡↔环面的螺线感接近),不同类之间加一个"短暂散开再聚合"的过渡动画

2. **骨架音密度**: 每小节 1 个骨架音太少(生成器完全失控),每拍 1 个太多(和手写没区别)
   - 建议: 每小节 2-3 个骨架音 (正拍 + 第三拍),中间留白给生成器

3. **和弦约束可能让音乐"太安全"**: V2 的五声音阶已经很安全了,再加和弦过滤可能失去冒险感
   - 缓解: 70% 的音走和弦内/五声音阶,30% 允许临时经过音(半音)

4. **性能**: 图案混合导致 target() 调两次,N=10000 时翻倍计算
   - 缓解: 只在乐章重叠的 4-5 秒内双算,其他时间单算
